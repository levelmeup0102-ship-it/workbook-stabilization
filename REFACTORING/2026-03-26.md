## 확정된 설계 사항

### 1. DELETE 엔드포인트 RESTful 재설계 (방안 A 확정)

```
DELETE /api/passages/books/{book_name}           → 교재 삭제
DELETE /api/passages/units/{book_name}/{unit}     → 단원 삭제
DELETE /api/passages/{book_name}/{unit}/{lesson}  → 강/과 삭제
```

- 3개 엔드포인트가 내부적으로 repository.delete() 공통 함수 1개를 재사용
- 기존 `DELETE /api/books` 제거 (passages로 통합)

### 2. DB 스키마 설계 확정

**passages 테이블:**
```
id(PK, auto increment) | book_name | unit | lesson(구 pid) | english_text | korean_translation | updated_at
```
- title 컬럼 삭제 → 프론트에서 `unit + lesson` 조합 표시
- korean_translation 컬럼 추가 → 업로드 시 영문+해석 분리 저장

**workbook_contents 테이블 (신규, step_cache 대체):**
```
id(PK) | passage_id(FK → passages.id) | stage_name | workbook_data(JSON) | updated_at
```
- passages.id 단일 FK로 관계 설정 (다중 PK FK 대신)
- workbook_data는 JSON 통째 저장 (PostgreSQL JSON 연산자로 다른 프로젝트에서 부분 조회 가능)

### 3. updated_at 기반 캐싱 전략 확정

```
워크북 생성 요청 시:
  1. pu_snapshot = passages.updated_at 저장
  2. Stage 1~8 실행 (비동기, 수 분 소요)
  3. 완료 시: pu_current = passages.updated_at 재조회
  4. 워크북은 항상 저장 (Claude API 비용 낭비 방지)
  5. 변경 시에만 응답에 passage_changed: true 포함 (미변경 시 필드 없음)
     - 프론트: "생성 중 지문이 수정되었습니다. 새 워크북을 다시 만드시겠습니까?" confirm
     - yes → 재생성 / no → 현재 워크북 사용
  - 추가 비용: DB 조회 2회 (시작 시 pu_snapshot, 완료 시 pu_current)
```

- 기존 파일 기반 캐싱(step*.json) 완전 제거
- 생성 중 지문 수정 시에도 안전하게 동작
